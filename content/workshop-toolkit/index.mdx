# Workshop Modes

import { Tabs, Callout, Steps } from 'nextra/components'

Designing agent experiences requires input from people who rarely think in terms of AI. The goal is not to ask stakeholders to design the agent, but to create conditions where their expertise surfaces naturally. Four frameworks anchor the conversation: the **Delegation Ladder** ([Parasuraman, Sheridan & Wickens, 2000](https://doi.org/10.1109/3468.844354)) places tasks on a spectrum from "Watch Me" to "Do It"; the **Trust Battery** ([Lee & See, 2004](https://doi.org/10.1518/hfes.46.1.50.30392)) tracks what charges and drains trust; the **Blast Radius Map** ([FMEA](https://asq.org/quality-resources/fmea)) sizes consequences into inner (reversible), middle (costly), and outer (irreversible) rings; and the **Information Asymmetry Grid** ([Johari Window](https://en.wikipedia.org/wiki/Johari_window)) maps what the agent knows vs what the user knows. Disagreement between stakeholders is the primary design input.

<Tabs items={['In-Person Workshop', 'Remote Workshop']}>
  <Tabs.Tab>
    ### In-Person Workshop

    <Callout>
    You are not mapping tasks to screens. You are mapping tasks to delegation levels, trust thresholds, and failure recovery paths. Stakeholders are designing for something they have never used, so you use indirect techniques to surface requirements. Disagreement is the design input, not something to resolve.
    </Callout>

    <Steps>

    ### Task Inventory + Delegation Placement
    Everyone writes tasks on sticky notes. Cluster by similarity, then place each on the **Delegation Ladder** poster. Where people place the same task at different levels, stop and discuss. Those disagreements reveal where the agent needs adaptive behaviour or configurable autonomy.

    ### Blast Radius Mapping
    Place key tasks on concentric circles drawn on the whiteboard. For each outer-ring task, walk through: *"The agent gets this wrong. What happens next? Who is affected? How bad does it get?"* This surfaces approval flow and guardrail requirements.

    ### Trust Battery Discussion
    For 3-4 key tasks, discuss what charges and drains the trust battery. Capture conditions: *"I'd trust it if it showed me its reasoning"*, *"I'd lose trust if it acted without asking on anything financial."* These become transparency and escalation design requirements.

    ### Failure Mode Storytelling
    Read a scenario aloud: *"The agent just did [something wrong]."* Each participant writes what went wrong, what the user should see, and what should happen next. Share and cluster by theme. This gives you error handling and recovery patterns.

    ### "Where's the Human?" Walkthrough
    Walk through a draft agent workflow on the whiteboard. At each step, participants place **green dots** (agent can act alone) or **red dots** (human checkpoint needed). For every red dot, capture what the human needs to see to decide.

    ### Red Team Round
    Ask: *"You are trying to get this agent to do something it shouldn't. How would you trick it?"* Domain stakeholders imagine more realistic abuse scenarios than security teams. Each scenario maps to a constraint or guardrail.

    ### Coordination Zone Mapping
    For each key task: *done with me* (collaborate closely), *done for me* (agent handles, human reviews), or *done under me* (agent works silently)? Different from the Delegation Ladder: the ladder asks *how much autonomy*, this asks *how visible should the agent be*. See [Coordination Zones](/experience-design/coordination-zones).

    ### Exception Storming
    Brainstorm every way a workflow could require human involvement, even when nothing has gone wrong. Categorise: Judgment, Authority, Ethics, Novelty, Conflict, Low confidence. Each category implies a different interface pattern. The exceptions *are* the UX. See [Agent Journey Mapping](/experience-design/agent-journey-mapping).

    ### Values Translation
    Stakeholders express what they care about in their own words: *"Be conservative with client money"*, *"Never send anything externally without checking with me."* Map each to a concrete agent boundary (approval gate, hard constraint, notification trigger, or autonomy limit). See [Core Principles](/design-principles/core-principles).

    ### Sludge Spotting
    Present the [six forms of agentic sludge](/design-principles/agentic-sludge). For each, ask: *"Where could this show up in our agent workflow?"* Domain experts will identify realistic scenarios where the agent could undermine user control. Each maps to a counter-pattern.

    </Steps>

    <Callout type="info" >
    **What you walk away with:** Tasks ranked by delegation readiness and risk · Trust thresholds per task type · Failure and edge case library · Human checkpoint map · Threat model grounded in domain expertise · Coordination zone assignments · Exception taxonomy · Value statements mapped to agent boundaries · Sludge risk assessment
    </Callout>

  </Tabs.Tab>
  <Tabs.Tab>
    ### Remote Workshop

    <Callout>
    Fewer activities, more facilitator-led. Silent writing replaces open discussion for most prompts (participants type before anyone shares, preventing anchoring). The facilitator pre-builds frameworks as digital templates so participants drag and drop rather than create from scratch.
    </Callout>

    <Steps>

    ### Delegation Ladder Placement
    The facilitator shares a digital board with the ladder. Participants silently drag tasks to the level where they'd be comfortable with an agent operating. Walk through the clusters, focusing on where placements diverge.

    ### Blast Radius Quick-Sort
    Present 10-15 key tasks one at a time. Participants vote: inner ring, middle ring, or outer ring. Capture split votes and probe: *"Three of you said outer ring, two said middle. What are you each imagining going wrong?"*

    ### Scenario Reactions
    Present 3-4 pre-written scenarios (*"The agent just..."*). Participants type responses: What went wrong? What should the user see? What should happen next? Synthesise live and pull out key themes.

    ### Trust Boundary Definition
    Walk through the Information Asymmetry Grid with domain-specific examples. For each quadrant, participants contribute. Focus on the bottom-right (neither knows) since that is where the strongest guardrails are needed.

    ### Coordination Zone Voting
    Present key tasks, participants vote: *done with me*, *done for me*, or *done under me*. Split votes reveal different comfort levels with agent presence. See [Coordination Zones](/experience-design/coordination-zones).

    ### Exception Brainstorm
    Participants silently type every way a workflow could require human involvement (judgment calls, authority questions, ethical tensions, novel situations). Cluster live and map each to an interface pattern. See [Agent Journey Mapping](/experience-design/agent-journey-mapping).

    ### Values Capture
    Each participant types 3-5 value statements: *"Never commit budget without my sign-off"*, *"Speed matters more than perfection for internal drafts."* Group into themes and map to agent boundaries. See [Core Principles](/design-principles/core-principles).

    </Steps>

    <Callout type="info">
    **What you walk away with:** Delegation levels with divergence points flagged · Risk-ranked task list · Failure scenarios and recovery patterns · Trust boundaries and transparency requirements · Coordination zone assignments · Exception taxonomy · Value statements mapped to agent boundaries
    </Callout>

  </Tabs.Tab>
</Tabs>

## Turning Workshop Output into Design Requirements

| Workshop Output | Design Requirement |
|---|---|
| Delegation Ladder placements | Autonomy levels per task type |
| Blast Radius positions | [Approval flow](/experience-design/approval-flow) requirements |
| Trust battery conditions | [Trust building](/design-principles/trust-building) and transparency patterns |
| Failure mode stories | Error handling and recovery flows |
| Red dot placements | Human-in-the-loop checkpoints |
| Red team scenarios | Constraint and guardrail specifications |
| Coordination zone assignments | [Coordination zone](/experience-design/coordination-zones) design per task |
| Exception taxonomy | Interface patterns per exception type |
| Value statements | Agent boundary rules in natural language |
| Sludge risk scenarios | Counter-patterns to prevent [agentic sludge](/design-principles/agentic-sludge) |
