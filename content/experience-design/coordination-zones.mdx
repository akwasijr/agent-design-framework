# Coordination Zones and Curves

## Moving Past the Binary

The "in the loop" vs "autonomous" framing is too coarse. Real workflows shift constantly between deep engagement and silent background work. Building on [Parasuraman, Sheridan & Wickens' automation model](https://doi.org/10.1109/3468.844354) (2000), we define **three coordination zones** and **coordination curves** that map how workflows move between them.

> The question is not whether the human is in the loop. It is how tightly the loop is drawn at any given moment.

## The Three Zones

Two dimensions define the spectrum: **AI salience** (how visible the AI is) and **human involvement** (how actively the person participates).

| Zone | AI Salience | Human Involvement | When to Use |
|---|---|---|---|
| **Done with me** | High | High | Complex decisions, creative work, high stakes, early trust-building |
| **Done for me** | High | Low | Well-defined tasks, research/synthesis, repetitive workflows |
| **Done under me** | Low | Low | Low-risk/high-frequency, personalisation, mature trusted systems |

**Zones are not fixed.** A single workflow moves between them. Example: an agent monitors data overnight (under me), compiles a draft report (for me), collaborates on interpreting a surprising trend (with me), then formats and distributes the final version (for me).

The design challenge is not picking a zone but designing smooth transitions between them.

## Coordination Curves

A coordination curve plots human involvement and AI salience over time through a workflow. The relationship between the two lines determines the active zone.

### Designing for the Valleys (Agent Works Independently)

Valleys are where trust is built or broken. The user cannot see what the agent is doing.

| Concern | Design Response |
|---|---|
| **Notification** | How does the user know the agent is working? |
| **Approval** | Are there checkpoints, or full autonomy until the next peak? |
| **Monitoring** | Can the user check in without disrupting? |
| **Auditing** | Can the user review what happened after? |
| **Recovery** | How does the user discover and correct mistakes? |

See [Ambient Awareness](/experience-design/ambient-awareness) for patterns that address valley design.

### Designing for the Peaks (High Collaboration)

- **Entry:** How does the user know it is time to engage? What context gets them up to speed?
- **Shared state:** Can both see the same information? Is reasoning visible?
- **Exit:** How does the user signal the agent can resume independent work?

### Designing for Transitions

Transitions are the highest-risk UX moments.

**Upward** (agent requests involvement): provide context first, explain why human judgment is needed, make deferral easy.

**Downward** (agent resumes independent work): confirm the handoff, set expectations, provide easy re-engagement.

## Responsive Salience

Rather than manual settings, the system calibrates its own salience from contextual signals:

| Signal | Effect |
|---|---|
| Higher task complexity or risk | Increases salience |
| Higher user expertise or trust history | Decreases salience |
| User is busy (focus mode, meeting) | Fewer interruptions |

Preferences diverge between individuals and do not correlate predictably with expertise. Responsive salience must respect individual preference, and those preferences must be easy to express and adjust. See Victor Yocco's [Smashing Magazine article](https://www.smashingmagazine.com/2026/02/designing-agentic-ai-practical-ux-patterns/) on supporting a spectrum of user comfort levels.

## Design Implications

1. **Match intensity to context.** Map workflows to identify which zones they move through. Do not apply one model uniformly.
2. **Build coordination curves into your design process.** Plot involvement and salience over time. Identify peaks, valleys, and transitions. Test whether transitions feel smooth.
3. **Support individual variation.** Let users temporarily increase salience ("walk me through this one") or decrease it ("just handle it") with lightweight controls.
4. **Design transitions deliberately.** Every transition needs context-setting, consent, and reversibility.
5. **Make the invisible auditable.** For every "done under me" or "done for me" period, users should be able to review what happened after the fact.
